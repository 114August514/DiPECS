# 第 11 次会议纪要：动作治理与安全边界

## 摘要

本次会议围绕动作执行安全展开。会议决定引入 `ActionLifecycle` 和 `AuthorizedAction`，明确决策后端只能提出建议，真正可执行的动作必须经过本地策略审查和 lifecycle seal。

## 会议信息

- **日期**：2026-05-27
- **议题**：动作治理、策略审查和 Android-safe action 边界
- **阶段**：动作层重构
- **参会人**：lsy, srh, ljs, wxz, sdd

## 背景

中期版本已有 `PolicyEngine` 和 `DefaultActionExecutor`，但动作治理还不够严格。尤其是当后续接入真实 LLM 时，如果模型输出直接映射到执行器，会有明显风险：

- 模型可能生成越权动作。
- 动作 target 可能不在当前上下文中。
- `PreWarmProcess`、`ReleaseMemory` 等名字容易被误解为系统级强动作。
- 执行失败、拒绝和回退需要可审计。

## 讨论内容

### 1. `SuggestedAction` 与 `AuthorizedAction` 分离

会议决定：

- `SuggestedAction` 是不可信建议。
- `PolicyEngine` 只对建议动作做裁决。
- `ActionLifecycle` 是 `AuthorizedAction` 的唯一构造点。
- action executor 只能接收 `AuthorizedAction`，不能接收模型输出。

这一设计避免了 agent 层绕过策略审查直接执行动作。

### 2. 策略检查项

`PolicyEngine` 需要覆盖：

- 后端能力等级：不同 route 有不同 action allow-list。
- 全局自动执行风险上限：默认只允许 `Low`。
- 置信度下限：低置信度直接拒绝。
- blocked action 子串。
- `Deferred` urgency 拒绝。
- 单 intent action 数量上限。
- target 是否出现在当前 `StructuredContext` 中。

常见拒绝原因包括 `RiskExceedsCapability`、`ActionCapabilityDenied`、`TargetNotInContext` 等。

### 3. 动作类型和 Android-safe 语义

会议确认当前动作集合保持保守：

| ActionType | 当前语义 |
| --- | --- |
| `NoOp` | 无副作用，占位或保守回退。 |
| `PrefetchFile` | 只允许 `url:https://...` 或 `uri:content://...`，写入 app cache。 |
| `KeepAlive` | 只调度 DiPECS-owned `JobScheduler` maintenance job。 |
| `ReleaseMemory` | 只清理 DiPECS-owned cache，如 `cache:prefetch` 或 `cache:all`。 |
| `PreWarmProcess` | `own:*` 预热自身资源；`pkg:*` / `notif:*` 只发用户可见提示。 |

会议特别强调：

- `PreWarmProcess` 不是静默启动第三方应用。
- `KeepAlive` 不是修改第三方进程保活。
- `ReleaseMemory` 不是清理第三方内存。

### 4. 审计状态机

每个动作建议需要形成确定性坐标：

```text
(window_ordinal, intent_ordinal, action_ordinal)
```

每个坐标最终恰好产出一条终态 `AuditRecord`。终态可以是成功、失败、schema 拒绝、policy 拒绝或 capability 拒绝。

## 决议

- 引入 `ActionLifecycle` 作为动作治理主入口。
- `AuthorizedAction` 字段私有化，action 层不能自行构造。
- 保留现有 5 类 `ActionType`，但 Android 执行语义收缩为安全子集。
- 所有动作必须有 audit record，方便 replay 和结题证据链。
