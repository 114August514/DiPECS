# 第 13 次会议纪要：Replay、Audit 与回归验证

## 摘要

本次会议聚焦验证体系。会议决定把 `aios-cli replay` 作为核心回归工具，支持 stage 输出、canonical audit stream 和 `audit_hash`，并将 daemon runtime trace 作为在线运行证据。

- **日期**：2026-06-10
- **议题**：Replay、audit hash、runtime trace 和端到端验证
- **阶段**：验证体系补强
- **参会人**：lsy, srh, ljs, wxz, sdd

## 背景

到本阶段，采集、脱敏、决策、策略和动作治理已经逐步成型。下一步需要解决“怎么证明链路稳定可复现”的问题。

评审和结题需要的证据不仅是“代码能跑”，还包括：

- 同一 trace 是否能稳定回放。
- 策略拒绝是否可解释。
- 隐私字段是否越过 Privacy Air-Gap。
- 动作执行、拒绝和失败是否都有审计记录。

## 讨论内容

### 1. `aios-cli replay` 阶段输出

会议决定 replay 工具按阶段输出：

| Stage | 含义 |
| --- | --- |
| `ingest` | 解析 JSONL `rawEvent`，进入 `RustCollectorIngress`。 |
| `sanitize` | 执行 `DefaultPrivacyAirGap`。 |
| `context` | 关闭窗口并输出 `StructuredContext`。 |
| `decision` | 执行 `DecisionRouter`。 |
| `policy` | 做策略裁决，不执行 adapter。 |
| `execute` | 运行 `ActionLifecycle` 和 `OfflineAdapter`。 |

这样可以定位问题发生在哪一层，而不是只看到最终动作结果。

### 2. Canonical audit 和 `audit_hash`

会议决定 replay 输出 canonical audit stream，并计算 SHA-256 hash。

为保证可重复，canonical projection 需要剥离 volatile 字段：

- UUID
- `window_id`
- `intent_id`
- `latency_us`
- 其他运行时随机或 wall-clock 相关字段

最终 `audit_hash` 用于 golden regression。只要输入 trace 和策略配置不变，hash 应该稳定。

### 3. `OfflineAdapter`

Replay 不应该访问真实 Android、网络或系统状态，因此执行阶段注入 `OfflineAdapter`。

`OfflineAdapter` 需要完整覆盖当前全部 `ActionType`：

- `NoOp`
- `PreWarmProcess`
- `PrefetchFile`
- `KeepAlive`
- `ReleaseMemory`

这样可以验证动作治理链路，而不产生真实副作用。

### 4. Runtime trace

会议决定 daemon 在线路径支持 `--trace-output`，记录窗口级 runtime trace。它用于演示在线运行，而 replay 用于回归和审计。

## 决议

- `aios-cli replay` 成为回归验证主工具。
- replay 支持 stage NDJSON、canonical audit 和 `audit_hash`。
- daemon 增加 runtime trace 输出，用于运行证据。
- golden tests 覆盖 sample、denial、privacy leak 等场景。
