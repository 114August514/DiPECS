## 背景

项目已经形成当前主链路：

```text
Android public API / daemon source
    -> CollectorEnvelope / RawEvent
    -> PrivacyAirGap
    -> StructuredContext
    -> DecisionRouter
    -> PolicyEngine
    -> AuthorizedAction
    -> Android action bridge / local executor trace
```

但仍有一些早期设想没有进入最终主线，例如 Skills 自迭代、真实 Binder/eBPF、fanotify/system image 和完整真机验证。

## 讨论内容

### 1. 结题报告口径

会议决定最终报告必须按当前实现描述，不再沿用早期“云端 Skills 主导”的口径。

需要强调：

- 当前默认本地优先。
- `CloudLlmBackend` 可选，默认关闭。
- `AccessibilityService` 是 screening source，不进生产 Rust schema。
- Binder/eBPF 和 fanotify 是预留路线，不产生真实事件。
- Android action bridge 只执行安全子集。

### 2. 验证证据

已具备的证据：

- Rust 单元测试和集成测试。
- replay fixtures。
- policy denial golden tests。
- privacy leak regression tests。
- mock-socket action-loop e2e。
- runtime trace 输出能力。

仍需补强：

- 真机 Android trace 样本。
- Android 权限授予和 adb forward 的截图。
- CI artifact 链接。
- action socket ping 和 execute 记录。
- emulator e2e 的稳定结果。

### 3. 文档状态标注

会议注意到早期文档中还存在历史表述，例如：

- `MockCloudProxy`
- Skills 自迭代
- ActionBus
- 云端大模型默认参与判断

这些内容在历史报告里可以保留，但当前实现文档必须明确指出它们不是最终主线。`current/overview.md` 应作为最新入口。

### 4. 期末周任务安排

由于 6 月 15 日到 6 月 30 日是期末周，会议决定不再安排大功能，只做：

- 文档一致性修复。
- 安全漏洞收敛。
- replay 和 audit 证据补充。
- 最终 slides 和报告整理。

## 决议

- 结题材料以 `current/` 文档为准。
- 早期 deliverables 作为历史阶段材料保留。
- 最终报告诚实列出真机验证、系统级采集和部分动作能力的限制。
- 期末周不新增大功能。
