# 第 10 次会议纪要：Android 采集入口与 JSONL 生产路径

## 摘要

本次会议集中讨论 Android 到 Rust 的生产入口。考虑到 JNI 直接接入调试成本较高，会议决定优先采用 append-only JSONL 作为 Android public API collector 和 Rust daemon 之间的稳定边界。

## 会议信息

- **日期**：2026-05-20
- **议题**：Android 采集入口、数据源筛选与 JSONL 协议
- **阶段**：Android ingress 实现
- **参会人**：lsy, srh, ljs, wxz, sdd

## 背景

中期纪要中列出的待解决问题之一是 “Kotlin → Rust JNI bridge”。但实际推进时发现，JNI 虽然更直接，但会带来较高调试成本：

- Android Studio、Rust target、NDK 和 daemon 启动链路耦合较强。
- 事件格式不稳定时，JNI 接口容易频繁变更。
- 对 replay 和离线验证不友好。

因此会议决定先使用 JSONL 打通 production ingress。

## 讨论内容

### 1. Android promoted sources

会议确认第一批进入生产 Rust 管线的 Android public API 来源：

| Android 来源 | Rust 事件 | 说明 |
| --- | --- | --- |
| `UsageStatsManager` | `AppTransition` / `ScreenState` | 记录前后台切换和屏幕状态。 |
| `NotificationListenerService` | `NotificationPosted` / `NotificationInteraction` | 记录通知出现和用户交互。 |
| `DeviceContext` heartbeat | `SystemState` | 记录网络、电量、充电、屏幕等基本状态。 |

这些来源属于 Android 官方公开 API，适合普通模拟器和设备验证。

### 2. AccessibilityService 降级为 screening source

AccessibilityService 能看到更细粒度的 UI 信息，但隐私和权限风险都更高。

会议决定：

- Android app 侧可以保留 Accessibility 预览和调研能力。
- 生产 JSONL 中若没有 Rust schema，则写 `rawEvent: null`。
- Rust ingress 跳过 `rawEvent: null` 行。
- 在没有更明确脱敏规则前，不把 Accessibility 纳入 promoted sources。

这个决定后来写入 `current/data-flow.md` 和 `current/overview.md`。

### 3. Append-only JSONL 入口

会议确定 Android collector 写入 app 私有目录下的：

```text
<app-private-files>/traces/actions.jsonl
```

每行是一个 `CollectorEvent`，其中 `rawEvent` 必须符合 `aios-spec::RawEvent` 外部标签格式，才进入 Rust 生产管线。

Rust 侧由 `AndroidJsonlTailer` 读取新增完整行，并保留 offset。文件截断或轮转时重置 offset。

### 4. JSONL 路线的取舍

优点：

- 调试简单，每一行都是可观察事件。
- 天然适合 replay 和 golden trace。
- Android 和 Rust 解耦，schema 稳定后再考虑 JNI 或 socket。
- 便于 CI 使用固定 trace 验证。

不足：

- 相比直接 socket 或 JNI，延迟更高。
- 需要处理文件轮转、截断和半行写入。
- 长期在线运行仍需评估 I/O 开销。

## 决议

- 生产入口优先采用 append-only JSONL。
- `UsageStatsManager`、`NotificationListenerService`、`DeviceContext` 作为 promoted sources。
- AccessibilityService 保留为 screening source。
- `dipecsd --android-trace-jsonl` 作为 daemon 接入 Android trace 的主要方式。
